# Hugo Migration Plan

Timestamp: 2026-06-29T20:34:08-05:00

## Purpose

Use Hugo to turn the repository into a usable public reference site while preserving the existing transcript curation workflow during migration.

The long-term goal is a Hugo-native site layout, because Hugo sections, layouts, data files, and assets work best when the repository follows Hugo's conventions. The migration should still happen in stages so the existing curated corpus is not moved before the build, search, and deployment path is proven:

- `src/transcripts/json/` remains the raw source of record.
- `src/transcripts/txt/` remains the default inspection surface for curation and repair.
- `docs/questions/` remains the curated Markdown Q&A corpus during the first migration phase only.
- `site/content/questions/` becomes the target curated Q&A location after the Hugo prototype works.
- Hugo generates a searchable, browsable static site from the curated question content.

Current corpus shape:

- 278 curated Markdown files under `docs/questions/`.
- 265 numbered episode pages.
- 13 non-numbered or special pages.

## Success Criteria

The first useful Hugo version should provide:

- A homepage that immediately functions as a reference/search surface, not a marketing page.
- A searchable question index across question text, answer summaries, episode titles, and livestream numbers.
- An episode browser generated from `src/live-stream-list.md`.
- A stable page for each curated Q&A file.
- Direct timestamp links that preserve the current YouTube `?t=` behavior.
- GitHub Pages deployment from a GitHub Actions artifact.
- No manual rewrite of hundreds of curated pages just to start using Hugo.

## Non-Goals

- Do not replace the transcript curation workflow.
- Do not bulk rewrite existing Q&A content unless a generator produces deterministic, reviewable output.
- Do not require a server, database, or external search service.
- Do not require npm or a JavaScript build pipeline for the initial version.
- Do not expose generated TXT transcripts as the primary public experience unless that is explicitly chosen later.

## Recommended Architecture

Use a dedicated Hugo site directory and treat it as the target public-site structure:

```text
site/
  hugo.yaml
  content/
    _index.md
    questions/              Target curated Q&A content location
    episodes/
    specials/
  data/
    episodes.json           Generated from src/live-stream-list.md
    questions.json          Generated from curated question pages
  assets/
    js/search.js
    css/site.css
  layouts/
    _default/
    questions/
scripts/
  Build-HugoSiteContent.ps1
```

The transitional first implementation is:

1. Keep `docs/questions/*.md` as the source corpus.
2. Add `scripts/Build-HugoSiteContent.ps1`.
3. The script parses `src/live-stream-list.md` and `docs/questions/*.md`.
4. The script writes generated Hugo content and data under `site/content/` and `site/data/`.
5. Hugo builds the static site from `site/`.

After that path works, migrate the curated Q&A source files from `docs/questions/` to `site/content/questions/` and update the repo docs, scripts, and Codex skills to use the Hugo-native path. At that point `site/content/questions/` should be the edited source, not a generated mirror.

Generated Hugo data can be either committed or ignored. For the first implementation, commit the generator and generated output until the build is trusted. Once stable, move generated data files to ignored build artifacts if the action regenerates them reliably. Do not keep both `docs/questions/` and `site/content/questions/` as competing hand-edited sources.

## Layout Migration Strategy

Hugo-native layout is the better end state, but moving the corpus first creates avoidable risk. Use a two-step migration:

### Step A: Compatibility Build

- Source: `docs/questions/*.md`.
- Generated output: `site/content/questions/*.md` and `site/data/*.json`.
- Purpose: prove Hugo layouts, search, metadata extraction, and GitHub Pages deployment.
- Rule: do not hand-edit generated `site/content/questions/*.md`.

### Step B: Canonical Hugo Content

- Source: `site/content/questions/*.md`.
- Generated output: `site/data/*.json` and the Hugo `public/` build artifact.
- Purpose: simplify the repo around Hugo conventions after the UX works.
- Rule: remove or archive `docs/questions/` once every reference, script, README section, AGENTS instruction, and Codex skill points to the Hugo content path.

Expected final source layout:

```text
src/
  live-stream-list.md
  transcripts/
    json/
    txt/
site/
  content/
    questions/
    episodes/
    specials/
  data/
  layouts/
  assets/
scripts/
```

This preserves the transcript source hierarchy while letting the public reference corpus live where Hugo expects content.

## Data Model

Generate structured data from the existing files instead of relying on manual front matter at first.

Episode fields:

- `number`
- `title`
- `slug`
- `youtube_url`
- `video_id`
- `question_page`
- `status`
- `is_numbered`
- `is_special`

Question row fields:

- `episode_number`
- `episode_title`
- `question_page`
- `time_label`
- `start_seconds`
- `video_url`
- `question`
- `short_answer`
- `row_index`

Derived fields:

- `question_count`
- `search_text`
- `topics` when manual tagging is added later
- `series` for numbered livestream, special stream, or Assassin's Creed side content

## Phase 1: Prototype Site Skeleton

Deliverables:

- Add `site/hugo.yaml`.
- Add minimal layouts for home, question page, episode list, and search page.
- Configure `baseURL` for the project GitHub Pages URL.
- Configure Hugo to render current table HTML safely. The existing Q&A tables contain raw HTML anchors, so Hugo Goldmark settings must account for that or the generator must convert anchors into Markdown links.
- Add a local command path:

```powershell
pwsh -NoProfile -File scripts/Build-HugoSiteContent.ps1
hugo --source site
```

Acceptance checks:

- Hugo builds without warnings that hide content.
- A sample numbered episode page renders with its table intact.
- A sample special page renders and is discoverable.
- Timestamp links still point to YouTube with correct `?t=` seconds.

## Phase 2: Deterministic Compatibility Generator

Build `scripts/Build-HugoSiteContent.ps1` to parse existing repo sources while the corpus still lives under `docs/questions/`.

Inputs:

- `src/live-stream-list.md`
- `docs/questions/*.md`

Outputs:

- `site/data/episodes.json`
- `site/data/questions.json`
- `site/content/questions/*.md`

Rules:

- Do not alter files under `docs/questions/`.
- Preserve question text and answer summaries exactly except for deterministic link normalization if needed.
- Use `src/live-stream-list.md` to associate numbered pages with titles, slugs, and video IDs.
- Treat non-numbered pages as special pages and preserve their filenames.
- Fail loudly on malformed table rows, missing files, or inconsistent timestamp links.

Acceptance checks:

- Generated file count matches `docs/questions/*.md`.
- Numbered generated pages match the 265 numbered source pages.
- Special generated pages match the 13 non-numbered source pages.
- Every generated question row has page, question, answer, time, and link fields.
- The generator exits nonzero on malformed tables.

Exit criteria for this phase:

- The generated Hugo content is good enough that a later move from `docs/questions/` to `site/content/questions/` is mostly path rewiring, not content redesign.

## Phase 3: Search UX

Start with a static JSON search index and a small client-side search script.

Initial search behavior:

- Search across questions, short answers, episode titles, and episode numbers.
- Return row-level results, not only page-level results.
- Show episode title, timestamp, question, short answer, and direct video link.
- Include a link to the full curated page.
- Support simple filters:
  - numbered episodes
  - special pages
  - episode number range

Implementation options:

- MVP: normalized substring search over generated JSON, no dependency.
- Later: vendored Fuse.js or Lunr if relevance ranking becomes important.

Acceptance checks:

- Searching `Ramesses`, `Exodus`, `Joseph`, and `chronology` returns relevant rows.
- Search works from a local static build without a backend.
- Search index size is acceptable for GitHub Pages.
- Empty searches and no-result states are handled cleanly.

## Phase 4: Reference Browsing

Add pages that support reference use without requiring search.

Pages:

- Home: search, latest numbered streams, special pages, and current corpus status.
- Episodes: sortable list of numbered livestreams.
- Questions: all generated Q&A pages.
- Specials: non-numbered pages grouped separately.
- About: short explanation of source data, curation limits, and verification workflow.

Useful metadata on list pages:

- episode number
- title
- question count
- YouTube link
- curated page link
- transcript availability status

Do not create topic pages until there is real topic metadata. Topic tags should be added only after a controlled tagging workflow exists.

## Phase 5: Canonical Content Migration

After the Hugo site is useful and deployment works locally, migrate the edited curated pages into Hugo's content tree.

Actions:

- Move ordinary curated pages from `docs/questions/` to `site/content/questions/`.
- Keep filenames stable unless Hugo section routing requires a deliberate permalink policy.
- Add front matter only if it is useful and can be generated deterministically from existing metadata.
- Update `README.md`, `AGENTS.md`, `.codex/skills/*`, and scripts to refer to `site/content/questions/`.
- Decide whether old `docs/questions/` paths need redirects, compatibility pages, or a one-time breaking change.

Acceptance checks:

- No curation instruction still tells agents to create new public Q&A pages under `docs/questions/`.
- Search and episode pages build from `site/content/questions/`.
- The corpus page count remains 278 total, 265 numbered, and 13 special pages unless content changed intentionally.
- Existing timestamp links remain intact after the move.

## Phase 6: GitHub Pages Deployment

Use GitHub Actions to build and deploy the Hugo artifact.

Recommended flow:

1. Check out the repo.
2. Install or configure Hugo.
3. Run the content generator.
4. Run `hugo --source site --gc --minify`.
5. Upload the generated static output as a Pages artifact.
6. Deploy with GitHub Pages.

The repository should no longer depend on `docs/index.html`; that legacy search page has already been removed.

Acceptance checks:

- The action builds on a clean checkout.
- The Pages artifact contains the Hugo output, not the source repo.
- Broken internal links fail the build or are reported.
- The deployed site works under the repository subpath.

## Phase 7: README And Contributor Updates

After the prototype works and especially after canonical content migration, update durable docs:

- `README.md`: explain the Hugo site, generated content, and local preview commands.
- `AGENTS.md`: clarify whether the active source path is still transitional `docs/questions/` or canonical `site/content/questions/`.
- `.codex/skills/transcript-to-md-reference/SKILL.md`: update the output path once the canonical Hugo content migration happens.

Avoid large durable-doc rewrites before the prototype proves the architecture.

## Validation Checklist

Before merging the Hugo migration:

- `git status --short`
- `git diff --check`
- generator build succeeds
- Hugo build succeeds
- source page count equals generated page count
- numbered page count equals generated numbered count
- special page count equals generated special count
- representative timestamp links match source `?t=` seconds
- representative question tables render correctly
- search returns row-level results
- after canonical migration, no durable instruction still points new page creation at `docs/questions/`
- GitHub Pages action succeeds

## Risks And Mitigations

Risk: raw HTML timestamp anchors may not render under Hugo defaults.

Mitigation: configure Hugo Goldmark rendering for trusted curated content, or transform anchors during generation.

Risk: search JSON becomes too large.

Mitigation: index only question rows and short answers first; omit full transcript text from the public search index.

Risk: generated metadata drifts from curated source pages.

Mitigation: make the generator deterministic and fail on mismatches instead of silently guessing.

Risk: special pages do not map cleanly to numbered livestream metadata.

Mitigation: model them as `is_special=true` and require explicit metadata overrides only where needed.

Risk: Hugo migration changes curation behavior.

Mitigation: during compatibility build, keep all transcript work in the existing JSON, TXT, and `docs/questions/` workflow. After canonical migration, keep JSON and TXT behavior unchanged while moving only the curated public Markdown destination to `site/content/questions/`.

Risk: `docs/questions/` and `site/content/questions/` both become hand-edited sources.

Mitigation: allow the duplicate tree only during compatibility generation. Once the Hugo layout is canonical, remove or archive `docs/questions/` and update every instruction path.

## Suggested First PR Scope

Keep the first PR small enough to review:

- Add `site/` skeleton.
- Add `scripts/Build-HugoSiteContent.ps1`.
- Generate a site from the existing corpus.
- Add static row-level search.
- Add a local build command.
- Do not add manual topic tagging.
- Do not move or rewrite `docs/questions/` in the first PR; schedule that as the canonical content migration after the Hugo build is proven.
